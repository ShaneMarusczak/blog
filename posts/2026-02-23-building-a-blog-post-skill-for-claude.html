Building a Blog Post Skill for Claude

2026-02-23 · ai-workflow

TL;DR: Claude's custom Skills feature lets you encode repeatable workflows. This post walks through building one that converts conversations into structured, standalone HTML blog posts using an expanded STAR framework — and the design decisions behind it.

Context

Claude conversations generate a lot of original thinking. Security research, debugging sessions, architecture discussions, theological deep-dives — whatever the domain, the ideas evaporate once the chat window closes. There's no built-in pipeline from "productive conversation" to "published artifact."

Claude's custom Skills solve exactly this kind of repeatable workflow gap. A Skill is a set of instructions packaged as a SKILL.md file that Claude reads at invocation time, giving it a consistent playbook for a specific task. The goal here was to build one that turns any substantive conversation into a publish-ready blog post.

Problem

Without a defined structure, asking Claude to "write this up as a blog post" produces inconsistent results. Sometimes it's too long. Sometimes it reads like a transcript. Sometimes it's generic content-marketing filler that no developer would finish reading. The output format varies, the voice drifts, and there's no quality floor.

The Skill needed to solve three things: give the output a consistent structure that serves developer readers, enforce a voice that sounds like a practitioner rather than a content farm, and produce files that can be published without post-processing.

Approach

The STAR framework (Situation, Task, Action, Result) is a well-known structure for case studies, but it's optimized for interview answers, not technical blog posts. It needed expansion. The final structure adds sections for the reasoning behind the approach, the messy middle of the process, source attribution, and open questions — things a real engineering write-up needs but STAR leaves out.

The expanded framework:

  • Context — why this matters right now
  • Problem — the specific gap or question
  • Approach — what was chosen and why
  • Process — the actual work, including failures
  • Sources & References — what informed the work
  • Outcome — what was achieved
  • Open Questions — what's unresolved

A mandatory TL;DR sits at the top. Developers scan before they read. If the first few seconds don't signal relevance, they leave.

Process

The Skill went through several iterations. The first draft used Markdown output with YAML front matter, assuming a static site generator like Jekyll or Hugo would process the files. This was changed to standalone HTML — a complete document with <head>, inline CSS, and semantic markup that renders directly in a browser. No build step, no dependencies. The site's index page links to the file and it just works.

I initially scoped the voice as "company wiki" — internal, utilitarian. That was wrong. The target is a public dev blog. The audience is other practitioners. The corrected register is "senior engineer sharing knowledge with the community." That distinction matters: wiki writing assumes a captive audience; blog writing has to earn attention.

A few specific decisions that came out of iteration:

First person is earned, not default. The primary voice is third-person analytical. First person appears only at decision points and pivots — "I noticed X, so I shifted to Y." This keeps the focus on the work rather than the author.

A pushback clause was added. If a conversation is too thin to support a real post, the Skill instructs Claude to say so rather than padding. It can suggest a shorter format like a TIL snippet instead. A blog post with nothing to say is worse than no blog post.

HTML escaping is called out explicitly. Code examples inside <pre><code> blocks need < rendered as &lt;, > as &gt;, and & as &amp;. Without this instruction, Claude occasionally outputs raw angle brackets that the browser interprets as tags, breaking the code display.

File naming is specified. Output follows YYYY-MM-DD-slug-title.html so files sort chronologically in a directory listing and the slug is human-readable.

Anthropic's Skill documentation imposes constraints that shaped the final version. The description field — which Claude uses to decide whether to invoke the Skill — has a 200-character maximum. The original description was well over that. Compressing it to 188 characters while retaining enough trigger phrases required careful word choice.

Sources & References

  • How to create custom Skills — Claude Help Center — official spec for Skill structure, metadata limits, and packaging
  • How to write a great technical blog post — freeCodeCamp — informed the TLDR and scanability decisions
  • Developer-Friendly Blog Post Structure — DevRel Bridge — reinforced the emphasis on working code examples and clear headings

Outcome

The finished Skill is a single SKILL.md file at 141 lines. It produces standalone HTML blog posts with consistent structure, enforced voice guidelines, proper code formatting, metadata for search engines, and a mandatory TL;DR. The workflow is: have a conversation, say "write this up", iterate on drafts, publish the final HTML file.

The Skill is generic — it doesn't assume a specific author, domain, or hosting platform. Any Claude user can drop it into their Skills directory and start producing posts from their conversations.

Open Questions

CSS coupling. The inline styles make posts readable standalone, but they'll clash with any site that has its own stylesheet. A future version might strip the inline CSS and assume the site handles presentation, or offer a flag to toggle between standalone and embedded modes.

Multi-post conversations. The Skill suggests splitting when a conversation covers multiple topics, but doesn't define how to handle that mechanically. Should it produce multiple files in one pass? Ask the user to pick which thread to write up first?

Syntax highlighting. The HTML uses class="language-xx" on code blocks, which libraries like Prism.js and highlight.js can pick up. But without one of those libraries loaded, code blocks are unstyled monospace. Worth considering whether to bundle a lightweight highlighter or leave that to the site.
